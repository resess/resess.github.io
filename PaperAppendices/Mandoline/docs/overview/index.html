Online Appendix

Step 1: Static Analysis.

MANDOLINE runs a lightweight static analysis to instrument the app and to calculate control-flow and dependence relationships for each statement. The instrumentation records a unique identifier for each basic-block and a unique identifier of the execution thread (using Thread.getId()). The user installs the instrumented app on the device, MANDOLINE either replays a pre-recorded execution script provided by the user or allows the user to trigger the app manually to reproduce its faulty behavior. The execution trace produced during this process, comprised of a sequence of the executed statement instances, is passed to the second step â€“ Dynamic Control-Flow Graph (DCFG) construction.

Step 2: DCFG Construction.

MANDOLINE first extracts all statement instances for each executed basic-block, producing a set of DCFG nodes. It then separates the nodes by their thread identifiers and adds control-flow edges between statement instances with the same thread identifier in their order of execution. We then identify and detect Android callback methods using the callback identification logic of FlowDroid. We add a registration dependency edge to represent each such registration. We also add a registration dependence edge between the statement that creates the thread and the first statement instance of its corresponding new thread. The thread identification logic is also borrowed from FlowDroid.

Step 3: Dynamic Slicing.

MANDOLINE slices the produced DCFG using the slicing criterion provided by the user: a tuple consisting of a statement instance to slice from and any number of variables used in the instance. It returns a slice: a set of statement instances that affect the variables used at the statement through transitive data and control dependencies. For control dependencies and data dependencies through local variables, we follow the classic slicing algorithm by Agrawal et al. (H. Agrawal, 1991). For a field variable v, we use field alias analysis to find the statements that define all fields that alias the variable v at statement s, then, among these definitions, find the definition that reaches the field v. Unlike static approaches, we perform the alias analysis directly on the execution trace. Our intuition is that the execution trace contains all assignments that cause two fields to alias one another. When the assignment that cause aliasing are inside the framework, our framework method modeling replaces the framework method calls by assignments that capture the aliasing effect of the method. We discuss how our framework method modeling works in detail after discussing the field alias analysis. Our analysis aims at finding the set of all statement instances that cause the field v to alias other fields. The last executed statement in the set that defines a variable that alias v is the statement whose definition reaches s.
