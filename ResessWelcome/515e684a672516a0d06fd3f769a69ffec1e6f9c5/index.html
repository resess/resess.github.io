    Scientific Writing

    • Each paper has four main parts of "identification": (1) title, (2) names of ALL authors, (3) publication venue, and (4) year. Always mention all these four parts when describing a paper
    • Conference papers can be published in the main/research track (typically, 10-12 pages), adjunct track (typically, 2-10 pages), or a collocated event, such as a workshop (can be any number of pages). Research track papers are expected to be of the highest quality and level of detail. Always mention the track if the paper is not published in the research track.
    • Summarize each and every paper you read, even in a few sentences, in the ReSeSS paper repository. Your future self will thank you, as details will become fuzzy after reading many papers.
    • A good summary should contain (i) a short description of the paper's main idea, (ii) a detailed description of the technical approach, (iii) your critical analysis of the strengths and weaknesses of the approach, which goes beyond those listed in the papers, and (iv) possible suggestions for improvements and follow-up work.
    Excellently written by Victor Cheng:


    Fuzzy writing often reflects fuzzy thinking. One of the best ways to improve your thinking is to articulate your thoughts in writing. When you write out your thoughts, you have something to analyze, revise, and improve.

    Editing = Clarifying your thinking . The more you write and edit, the clearer your thinking becomes.
    Here are three tips for clarifying your writing/thinking:

    1. Structure your Thoughts.
      When possible, group similar ideas together into distinct categories of information. One of the easiest ways to confuse an audience is to not organize your thoughts into categories. The second easiest way to confuse an audience is to violate your own categorization structure.


      For example, there are three steps to riding a bicycle:
      a) starting
      b) coasting
      c) stopping

      This is a clear structure. Explaining the concept of pedals clearly belongs in the first category on “starting.” Explaining the role of brakes clearly belongs in the last category on "stopping." If you discuss brakes during starting, and pedals during stopping (assuming you’re using handbrakes), the content doesn’t fit the structure. Readers find this confusing. In addition, let's say your written work includes the following sentence: “Don’t forget to oil your bike chain regularly to ensure proper functioning.” This comment doesn’t fit your structure. It sticks out.

      This leads me to my second tip:

    2. Stay Focused by Eliminating Extraneous Ideas.
      Quite often, less is more. It is better to convey an idea simply and cleanly than to cram more information in that doesn’t fit the focus of your written document.

    3. Be Concise by Editing Ruthlessly.
      As you review your work, ask yourself the following question (over and over again): Can I eliminate this word, phrase, or paragraph without changing the meaning? If so, delete it. Clear writing comes from getting your key ideas across while using not one word more or less than necessary. Clear thinking comes from focusing on the single most important crux of your thought -- rather than swimming in a sea of jumbled ideas. The better you get at editing your writing, the better you will automatically get at thinking more concisely.

    Technical Tips and Tricks

    • Two options:
      • GNU Screen
      • Tmux
    • GNU Screen
      • Start screen: screen
      • Detach screen: CTRL-A-D
      • Re-attach latest screen: screen -r
      • Re-attach screen by ID: screen -r [ID]
      • List screen sessions: screen -ls
    • Tmux
      • Start tmux session: tmux new -s [TAG_NAME]
      • Access existing tmux by ID: tmux a [ID]
      • Access existing tmux by tag name: tmux a -t [TAG_NAME]
      • List tmux sessions: tmux ls
      • Kill tmux session: tmux kill-session -t [TAG_NAME]

    On Zeus

    1. Include the following in your PATH variable, if not there already:
      /emulator
      /platform-tools/
      /cmdline-tools/tools/bin

      E.g., use the following command:
      export PATH=$PATH:sdk_path/emulator/

    2. Check permissions of /dev/kvm. If it does not have crwxrwxrwx+ permissions, give it to the file using the following command:
      sudo chmod 777 /dev/kvm
    3. Create a VNC server and write down its id.
      vncserver (id is after "... desktop is zeus:")
    4. Find the IP address, and write it down.
      ifconfig (The IP address is inet addr for eth0)

    On Your Machine

    1. Install VNC viewer if not done already:
      https://www.realvnc.com/en/connect/download/viewer/
    2. Make sure you are connected to UBC network (ubcprivate or via VPN)
    3. Open VNC viewer, sign up or sign in.
    4. Open the new connection window:
      File->New connection->General
    5. Set the VNC server to be Zeus.
      ip_address:590 + id of VNC server written down in Step 3.
    6. Double click to connect to the viewer.
    7. Input VNC password
    8. You are ready to go. For example, open a terminal window to run any command:
      Right click->Debian->Applications->Terminal Emulators->LXTerminal
    1. Follow the steps in Section "Setting Up a VNC Viewer on Zeus" to set up VNC viewer
    2. Open VNC viewer, connect to Zeus, and open a terminal window
    3. On Zeus, create an emulator and write down its name:
      avdmanager create avd -n test -k "sdk_id"
    4. Create a script, android_emu.sh, with the following code and make the script executable.
      #!/bin/bash
      adb kill-server
      adb start-server
      emulator -avd $1 -port 5554 -gpu guest -writable-system -no-snapshot
    5. Run the script created in Step 4, use the emulator name written down in Step 3 (it is passed to the script as the first parameter). For example,
      ./android_emu.sh test
    6. Wait and you should see the emulator!
